# Skill 迭代六维分析框架

## 概述

当 Agent 对比用户修改前后的版本时，必须从以下六个维度给出结构化分析，而不是笼统说"改得更好"。

## 六维分析框架

### 1. 开头方式
- 修改前：是陈述事实，还是制造冲突/挑战假设？
- 修改后：是陈述事实，还是制造冲突/挑战假设？
- 改动本质：从什么变成了什么？
- 用户可能的意图：为什么不喜欢原来的开头？

### 2. 概念处理
- 修改前：是否使用了大词、模糊概念、行业黑话？
- 修改后：是否重新定义了某个常见概念？是否把大词换成了具体定义？
- 改动本质：从什么变成了什么？
- 用户可能的意图：为什么不喜欢原来的概念处理方式？

### 3. 抽象 vs 具体
- 修改前：是否用抽象概括（"很多企业""提升效率"）？
- 修改后：是否用具体数字、场景、案例替代了抽象概括？
- 改动本质：从什么变成了什么？
- 用户可能的意图：为什么需要更具体？

### 4. 叙事视角
- 修改前：是第一人称亲历，还是第三方观察？
- 修改后：是第一人称亲历，还是第三方观察？
- 改动本质：从什么变成了什么？
- 用户可能的意图：为什么需要这个视角？

### 5. 情绪节奏
- 修改前：是否有起伏（冲突→解决→升华）？还是平铺直叙？
- 修改后：是否有起伏？节奏如何变化？
- 改动本质：从什么变成了什么？
- 用户可能的意图：为什么需要这个情绪节奏？

### 6. 结尾力度
- 修改前：是在描述状态，还是在召唤行动/给出金句？
- 修改后：是在描述状态，还是在召唤行动/给出金句？
- 改动本质：从什么变成了什么？
- 用户可能的意图：为什么需要这个结尾？

## 输出格式

每个维度的输出必须包含：

```markdown
### [维度名]
- **修改前**：[原文摘录]
- **修改后**：[修改后摘录]
- **改动本质**：[一句话总结]
- **用户意图推测**：[为什么这样改？]
```

## 总结部分

六维分析完成后，必须给出：

1. **用户核心偏好总结**：用户最在意什么？
2. **Bad case 清单**：记录"之前这样写，用户改成了那样"的具体案例
3. **Skill 优化建议**：基于以上分析，具体建议修改 SKILL.md 的哪些部分

## 迭代原则

- 对比要具体
- 分析要结构化
- 推断用户意图
- 保留 bad case
- 区分平台（飞书 CLI 只是写入渠道）
- 用户确认（修改前展示草案）
